# 简历解析加载态与 Markdown 展示改造设计文档

## 1. 背景

当前简历诊断助手已经具备如下能力：

- 上传 PDF 简历
- 使用 `MarkItDown` 生成原始 Markdown
- 使用小模型整理 Markdown，并在失败时自动降级
- 将最终 Markdown 继续用于后续分析
- 在结果页展示 Markdown 结果

但当前体验还存在 3 个明显问题：

1. 当后台在执行 LLM 规范化时，前端缺少明确反馈，用户容易误以为页面无响应。
2. 结果页里的 Markdown 现在更接近原始文本展示，缺乏富文本层次，视觉效果一般。
3. 当前默认模型速度偏慢，需要切换到更快的模型。

## 2. 目标

本次改动只解决以下 3 个问题：

1. 上传后提供明确的整页加载态。
2. 将 Markdown 改为双栏富文本对比展示。
3. 将默认模型切换为 `XiaomiMiMo/MiMo-V2-Flash:xiaomi`。

## 3. 非目标

本次改动明确不做以下内容：

- 不调整评分规则
- 不重写 `ResumeSectionParser` 的整体解析策略
- 不增加模型切换器
- 不将 Markdown 展示改造成完整的简历预览器
- 不引入前端框架或复杂异步前端架构

## 4. 用户已确认的关键约束

- 上传后使用 `整页加载态`
- Markdown 展示采用 `双栏对比风`
- 模型 `完全切换` 到 `XiaomiMiMo/MiMo-V2-Flash:xiaomi`

## 5. 方案概览

本次改造分为 3 个部分：

1. **加载态改造**
   用户提交表单后，立即进入一个独立的处理中页面。

2. **Markdown 展示改造**
   结果页中将原始 Markdown 和整理后 Markdown 以双栏富文本形式展示。

3. **模型切换**
   将 Markdown 规范化默认模型由当前模型切换为 `XiaomiMiMo/MiMo-V2-Flash:xiaomi`。

## 6. 加载态设计

### 6.1 交互方式

当前页面是服务端渲染流程，因此本次采用最稳妥的方式：

- 用户点击上传按钮后
- 页面立即跳转到一个独立的处理中页面
- 处理中页面展示状态说明和动效
- 后端继续执行解析、规范化和分析流程

### 6.2 页面目标

加载页的目标不是展示真实进度，而是向用户明确传递：

- 当前正在工作
- 当前工作包含多个阶段
- 请稍等，不要重复提交

### 6.3 展示内容

加载页建议展示以下信息：

- 标题：正在解析简历
- 副标题：系统正在提取 PDF、整理 Markdown 并生成诊断结果
- 3 条阶段提示：
  - 正在提取 PDF 内容
  - 正在整理 Markdown 结构
  - 正在生成诊断与评分

### 6.4 技术实现方向

由于当前还是传统表单提交流程，本次不引入复杂轮询或异步任务系统。

推荐实现方式：

- 提交按钮触发后，前端立即展示一个全屏 loading overlay
- loading overlay 的视觉效果要足够明确，看起来像进入了独立处理中状态
- 表单按钮禁用，避免重复提交

说明：

虽然用户希望的是“整页加载态”，但在当前同步请求架构下，更实际的落地方式是：

- 保持表单直接提交
- 在提交瞬间用全屏遮罩模拟整页处理中状态

这能在不改动后端请求模型的前提下实现接近“独立加载页”的体验。

## 7. Markdown 展示设计

### 7.1 展示目标

Markdown 展示不再是单栏文本块，而要变成便于人工审核的对比视图。

核心目标：

- 一眼看到原始提取质量
- 一眼看到模型整理后的结构变化
- 展示形式尽量接近富文本，而不是裸文本

### 7.2 布局方式

采用双栏布局：

- 左栏：原始提取 Markdown
- 右栏：整理后 Markdown

桌面端：

- 左右并排

移动端：

- 上下堆叠

### 7.3 渲染方式

为了获得“富文本”效果，展示不再直接使用纯 `pre`。

建议方式：

- 保留原始 Markdown 字符串
- 后端将 Markdown 渲染成 HTML
- 模板中展示渲染后的 HTML 结果

这意味着页面里要展示的是：

- 原始 Markdown 的渲染结果
- 整理后 Markdown 的渲染结果

而不是单纯打印 Markdown 源码。

### 7.4 展示内容

每一栏建议包含：

- 区块标题
- 一段说明文字
- 渲染后的富文本正文

整体区域上方保留：

- 是否发生降级
- 降级原因

### 7.5 视觉方向

该区域应偏“文档对比视图”，不是控制台。

建议视觉特征：

- 清晰边框卡片
- 标题层级明确
- 段落和列表有舒服的留白
- 左右对比在视觉上平衡

## 8. 模型切换设计

### 8.1 默认模型切换

当前 Markdown 规范化默认模型将直接切换为：

`XiaomiMiMo/MiMo-V2-Flash:xiaomi`

### 8.2 配置策略

根据用户确认，本次不保留“默认旧模型 + 可配置覆盖”的设计。

即：

- 默认模型直接改为新模型
- 当前代码中对应默认配置同步更新

### 8.3 保持不变的部分

以下调用机制保持不变：

- 仍通过 `MODELSCOPE_API_KEY` 从环境变量读取密钥
- 仍通过 ModelScope OpenAI 兼容接口调用
- 仍保留缺 key 自动降级
- 仍保留模型失败自动降级

## 9. 对现有代码的影响

预计会影响以下文件：

- `app/config.py`
  - 切换默认模型配置
- `app/services/markdown_normalizer.py`
  - 继续复用现有调用逻辑，只更换默认模型
- `app/templates/index.html`
  - 增加上传提交后的 loading 结构
- `app/static/styles.css`
  - 增加加载态样式和 Markdown 双栏展示样式
- `app/templates/result.html`
  - 将 Markdown 展示改成双栏富文本渲染
- `app/routes/web.py`
  - 可能需要将 Markdown 渲染后的 HTML 一并传给模板

## 10. 错误处理要求

本次改动不改变原有错误边界：

- PDF 提取失败：仍然返回 `400`
- Markdown 规范化失败：仍然自动降级
- 缺少 API Key：仍然自动降级

加载态只是体验改造，不应改变后端错误语义。

## 11. 验收标准

当以下条件全部满足时，本次改动可视为完成：

1. 用户点击上传后，页面能立即进入明确的处理中状态。
2. 处理中状态下，用户能感知后台正在执行多个阶段。
3. 结果页中原始 Markdown 和整理后 Markdown 采用双栏展示。
4. Markdown 展示为富文本效果，而不是单纯 `pre` 文本。
5. 默认模型已经切换为 `XiaomiMiMo/MiMo-V2-Flash:xiaomi`。
6. 缺少 `MODELSCOPE_API_KEY` 时，链路仍能自动降级。
7. 相关测试覆盖上传态和结果页展示回归。

## 12. 当前结论

本次改动采用以下实现方向：

- 上传后使用整页感知明确的 loading 状态
- 结果页中使用双栏富文本方式展示原始与整理后 Markdown
- 默认模型直接切换到 `XiaomiMiMo/MiMo-V2-Flash:xiaomi`

该设计在不显著扩大当前 MVP 架构复杂度的前提下，可以明显改善用户等待感知、Markdown 结果可读性，以及整体使用体验。
